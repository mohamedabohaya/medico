# تحليل تطبيق MediGo

التطبيق الحالي عبارة عن واجهات UI لتطبيق طبي يحتوي على قسم مريض وقسم طبيب.

## الأطراف الرئيسية
1. المريض: يسجل، يبحث عن تخصص، يختار طبيب، يحجز خدمة، يسأل طبيب، يقيم الخدمة.
2. الطبيب: يسجل طلب انضمام، ينتظر الموافقة، يكمل بيانات العيادة والخدمات، يتابع الحجوزات والأسئلة والمعاملات.
3. الأدمن: مطلوب مستقبلاً للموافقة على الأطباء وتغيير حالة الطلب من pending إلى approved أو rejected.

## الخدمات الطبية
- زيارة عيادة.
- زيارة منزلية.
- مكالمة هاتفية.

كل طبيب يستطيع تفعيل خدمة أو أكثر حسب قدرته.

## البيانات التي يجب تخزينها
- بيانات المستخدمين المرضى.
- طلبات الأطباء قبل الموافقة.
- بيانات الطبيب بعد الموافقة.
- بيانات العيادة والعنوان.
- الخدمات المتاحة للطبيب.
- الأسعار.
- التخصصات الرئيسية والفرعية.
- الحجوزات.
- أسئلة المرضى.
- التقييمات.
- المعاملات.

## لماذا Firebase مناسب هنا؟
Firebase مناسب جداً لأن التطبيق يحتاج قاعدة بيانات مشتركة بين أكثر من جهاز: طبيب، مريض، وأدمن. SQLite لا يكفي وحده لأنه محلي على الجهاز ولا يشارك البيانات بين المستخدمين.

## نموذج قاعدة البيانات المقترح
### users
بيانات المرضى وحساباتهم.

### doctorRequests
طلبات تسجيل الأطباء، وحالتها: pending / approved / rejected.

### doctors
ملف الطبيب المعتمد وبيانات العيادة والخدمات والأسعار.

### appointments
الحجوزات بين المريض والطبيب ونوع الخدمة.

### questions
أسئلة المرضى المرتبطة بالتخصص أو الطبيب.

### ratings
تقييمات المرضى للأطباء.

### transactions
المعاملات المالية أو سجل الخدمات.
